# AI Browser Agent — Writeup

## Что это

Автономный AI-агент, который управляет браузером по текстовой команде пользователя.
Аналог Claude in Chrome: пользователь пишет задачу → агент сам открывает браузер,
кликает, заполняет формы, ищет товары, проверяет цены → возвращает результат.

Агент не знает заранее ни одного шага. Он смотрит на страницу через скриншот и DOM
и решает что делать — как человек.

---

## Что реализовано

### Ядро агента

**`main.py`** — точка входа. REPL-интерфейс: принимает задачи в цикле,
поддерживает аргумент командной строки (`python main.py "найди яблоки"`).
Управляет жизненным циклом браузера.

**`agent.py`** — конфигурация агента и системный промпт. Задаёт правила поведения:
формат рассуждений, порядок действий, проверку цен, ограничение на hover перед кликом.
Подключает lifecycle хуки (on_step_start / on_step_end).

**`browser.py`** — настройка браузера через browser-use. Видимый режим (headless=False),
подсветка элементов для vision, persistent sessions через cookies_file.

**`llm.py`** — инициализация LLM. Основной агент: GPT-4o с vision.
DOM Sub-agent: GPT-4o-mini для точечных вопросов про страницу без нагрузки на контекст.

**`context.py`** — управление историей шагов. Хранит только последние 10 шагов
(скользящее окно), отслеживает consecutive failures, детектирует застревание (3+ ошибки).

**`tools.py`** — кастомные инструменты поверх browser-use:
- `ask_human` — агент задаёт вопрос пользователю в терминале
- `confirm_action` — запрашивает явное разрешение перед опасным действием

### Паттерн 1 — Recovery (восстановление при ошибках)

Реализован на двух уровнях:

1. **browser-use `max_failures=3`** — встроенный механизм: при N ошибках подряд
   LLM получает сигнал переосмыслить подход и попробовать другую стратегию.

2. **`recovery.py` + `context.py`** — поверх browser-use: ContextManager считает
   consecutive failures, при достижении порога `log_stuck()` логирует ситуацию
   и выводит пользователю сообщение `[Recovery] Агент адаптируется...`.
   `on_step_end` хук в agent.py отображает прогресс адаптации в реальном времени.

**`RECOVERY_HINT`** в recovery.py — текст-подсказка, который можно добавить
в системный промпт при необходимости (альтернативные подходы, перезагрузка, ask_human).

### Паттерн 2 — Security Layer (защита от деструктивных действий)

Реализован на трёх уровнях:

1. **Проверка задачи до запуска** (`main.py: is_destructive_task`) — если задача
   содержит слова "оплати", "удали", "купи", "подтверди" и т.д. — агент не стартует
   без явного подтверждения пользователя.

2. **URL-перехват на каждом шаге** (`security.py: is_destructive_url`) — хук
   `on_step_start` проверяет текущий URL агента. Если URL содержит `checkout`,
   `payment`, `confirm`, `удал` и т.д. — останавливает агент и спрашивает
   пользователя `"Продолжить? (да/нет)"`.

3. **Инструмент `confirm_action`** (`tools.py`) — агент сам может вызвать
   подтверждение когда считает действие опасным. Встроен в системный промпт
   как обязательное правило для оплат, удалений, отправки форм.

### Управление контекстом

- Агент получает только интерактивные элементы страницы (browser-use извлекает их сам),
  а не весь HTML — предотвращает переполнение токенов.
- История ограничена 10 последними шагами (`MAX_STEPS_IN_HISTORY`).
- Лимит входных токенов на запрос: `max_input_tokens=32000`.
- `enable_memory=False` — отключена долгосрочная память, только sliding window.

### Persistent Sessions

Сессия браузера сохраняется в `browser_profile/session.json` (cookies).
При первом запуске открывается браузер для ручного входа, после чего сессия
сохраняется и переиспользуется во всех следующих запусках автоматически.

## Средняя стоимость одного запроса

Из реального запуска (задача на Яндекс Лавке, 9 шагов):

```
Total input tokens (approximate): 44 345
```

### Расчёт по тарифам GPT-4o (актуальные цены):

| Статья                  | Токены        | Цена          |
|-------------------------|---------------|---------------|
| Input tokens (GPT-4o)   | ~44 000       | $0.110        |
| Output tokens (GPT-4o)  | ~2 000–4 000  | $0.020–0.040  |
| DOM Sub-agent (4o-mini) | ~2 000–5 000  | $0.001–0.003  |
| **Итого на задачу**     |               | **~$0.13–0.15** |

На один шаг агента: ~4 900 input токенов — это ожидаемо для vision-агента,
который каждый раз получает скриншот + DOM + историю.

Для сравнения: простая задача (3–5 шагов) обойдётся в ~$0.05, сложная (20+ шагов) — ~$0.40.

## Как создавался проект

### Первый подход — с нуля

Прочитал ТЗ, изучил всё что там было. Начал строить агента с нуля на чистом
Playwright + OpenAI API. Агент не работал нормально: постоянные ошибки с DOM-парсингом,
retry логика написанная руками, скриншоты отправлялись неправильно — в итоге агент
работал очень плохо и непредсказуемо.

### Решение — найти скелет

Решил не изобретать велосипед и поискать готовую основу. Нашёл на GitHub `browser-use` —
open-source библиотека именно для AI browser агентов. Изучил репозиторий, разобрался
в API, понял что это идеально подходит под требования ТЗ: встроенный DOM-экстрактор,
vision, tool calling, lifecycle хуки, persistent sessions — всё уже есть.

Принял решение: использовать browser-use как скелет и строить проект поверх него.
Это не упрощение — это именно то, что написано в ТЗ ("browser-use — готовая основа агента").

### Разработка с Claude Code

После того как определился со стеком — работал через детальный промпт и Claude Code агента.
Это позволило быстро итерировать: описывал что нужно, агент предлагал реализацию,
я тестировал на реальных задачах (Яндекс Лавка, hh.ru, habr), смотрел где ломается,
фиксил промпт системного агента.

### Сколько времени

На задание в общей сложности ушло около **15 часов** — изучение ТЗ, первый провальный
подход с нуля, изучение browser-use, архитектура, реализация, отладка на тестах.

Задание произвело азарт — сидел без остановки.

## Тесты и затраты

На тестирование потратил около **$8**. GPT-4o с vision — дорогая модель, каждый шаг
агента ~4 900 токенов только на input. Бесплатные лимиты заканчиваются моментально,
не успевая даже начать нормальное тестирование.

Без возможности тестировать продолжать разработку физически невозможно — агент
работает с живым браузером, каждый запуск стоит денег, и баги видны только на реальных
сайтах.

## Честная оценка проекта

Проект максимально сырой. Основные проблемы выявленные на тестах:

**Галлюцинации агента** — иногда встречаются. Агент видит скриншот и принимает
неверное решение: добавляет не тот товар (мандарины вместо яблок), дважды добавляет
один товар, считает задачу выполненной когда она не выполнена. Это фундаментальная
проблема vision LLM, а не только нашего агента — GPT-4o иногда "видит" то, что
ожидает увидеть.

**Что нужно для нормальной работы:**
- Больше тестов на реальных сайтах с разными сценариями
- Фикс конкретных паттернов галлюцинаций в системном промпте
- Возможно — верификационный слой (проверять корзину через отдельный LLM вызов
  после каждого добавления товара)
- Для production — переход на более надёжную модель или fine-tuning

Для MVP и демонстрации концепции — работает. Для реального использования нужна
ещё неделя тестов и итераций.

## Как пришёл к архитектуре

### Шаг 1 — выбор стека

Задание требовало: автономный агент, vision, tool calling, persistent sessions,
без хардкодных шагов. Рассматривал два варианта:

- **Писать с нуля** на Playwright + OpenAI API: полный контроль, но много boilerplate —
  нужно самому реализовывать скриншоты, DOM-парсинг, retry логику, форматирование
  элементов для LLM.

- **browser-use** как основа: готовый агентный цикл, встроенный DOM-экстрактор
  (только интерактивные элементы), vision, tool calling, lifecycle хуки.
  Остаётся добавить только бизнес-логику поверх.

Выбрал browser-use. Это не упрощение задачи — это правильное использование
готовой основы, как требует задание ("browser-use — готовая основа агента").

### Шаг 2 — архитектура файлов

Разбил по ответственностям из ТЗ, каждый файл делает одно:
- `browser.py` — только конфигурация браузера
- `llm.py` — только инициализация LLM
- `agent.py` — только системный промпт и хуки
- `tools.py` — только кастомные инструменты
- `context.py` — только история шагов
- `security.py` — только проверки безопасности
- `recovery.py` — только логика восстановления
- `main.py` — только точка входа и REPL

### Шаг 3 — реализация паттернов

**Security** — понял что нужно три эшелона: до запуска (задача), во время работы
(URL-хук), и по воле агента (инструмент). Только URL-хука недостаточно — агент может
нажать кнопку "Оплатить" без навигации на новый URL.

**Recovery** — browser-use уже делает retry через `max_failures`. Моя задача —
добавить видимость: пользователь должен видеть что агент не завис, а адаптируется.
Поэтому `log_stuck` и `[Recovery]` сообщения в терминале.

### Шаг 4 — отладка по ходу

Несколько проблем выявленных на тестах и исправленных в промпте:

1. **Hover перед кликом** — на Лавке кнопка "+" не видна без наведения. Добавил
   правило 8 в системный промпт.

2. **Зацикливание на одном элементе** — агент мог кликать один индекс бесконечно.
   Добавил правило 9: если одно действие дважды не дало прогресса — стоп.

3. **Лишние шаги после выполнения** — агент иногда шёл к оформлению заказа после
   добавления в корзину. Добавил явный запрет в правиле 10.

4. **Проверка цены** — агент видел цену одного товара, добавлял другой.
   Добавил правило 11: перед кликом "+" убедись что цена именно этой карточки
   соответствует условию.

## Как удешевить запросы

### 1. Заменить модель — самый большой эффект

Сейчас: `gpt-4o` на каждом шаге (~$2.50 за 1M input токенов).

| Модель | Input | Output | Разница |
|---|---|---|---|
| gpt-4o (сейчас) | $2.50/1M | $10/1M | baseline |
| gpt-4o-mini | $0.15/1M | $0.60/1M | **~17x дешевле** |
| claude-haiku-4-5 | $0.80/1M | $4/1M | ~3x дешевле |
| claude-sonnet-4-6 | $3/1M | $15/1M | сопоставимо |

Для простых задач (поиск, клики, добавление в корзину) `gpt-4o-mini` справляется.
Упасть в качестве можно, но для e-commerce задач — вполне работает.

### 2. Снизить `max_input_tokens`

Сейчас стоит `32_000`. Реальный средний запрос на 9 шагах — ~4 900 токенов на шаг.
Можно опустить до `16_000` или даже `8_000` — агент просто будет видеть меньше истории.

### 3. Уменьшить `MAX_STEPS_IN_HISTORY`

Сейчас `10`. Поставить `5` — история короче, меньше токенов в каждом запросе.

### 4. Отключить vision на простых шагах

`use_vision=True` — каждый скриншот добавляет ~1 000–3 000 токенов.
Можно переключить на `use_vision="auto"` или `False` — агент будет работать только с DOM.
Теряется понимание визуального контекста, но на 30–50% дешевле.

### 5. DOM Sub-agent и так дешёвый

`gpt-4o-mini` для `ask_about_dom` — уже сделано, это правильно.

### Итог — сколько можно сэкономить

Если заменить `gpt-4o` → `gpt-4o-mini` + снизить history до 5 шагов + `use_vision="auto"`:

- Сейчас: ~$0.13–0.15 за задачу
- После: ~$0.01–0.02 за задачу → **в 8–15 раз дешевле**

Компромисс: на сложных задачах с динамическим интерфейсом (попапы, анимации) качество упадёт.
Оптимально — **gpt-4o-mini для простых задач, gpt-4o только если агент застрял** (adaptive model switching).

## Запуск

```bash
cd agent
pip install -r requirements.txt
playwright install chromium
cp .env.example .env
# добавить OPENAI_API_KEY в .env

python main.py
# или сразу с задачей:
python main.py "зайди на habr.com и найди 3 статьи про AI-агентов"
```

При первом запуске откроется браузер для ручного входа на нужные сайты.
После нажатия Enter сессия сохраняется и используется автоматически.
